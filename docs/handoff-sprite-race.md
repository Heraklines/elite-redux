## Hypotheses tried and ruled out

- **Atlas JSON format wrong (w/h vs width/height)**: Was a real bug, fixed pre-handoff. Atlases now load when given time.
- **Phaser loader event timing wrong**: Tried per-file, generic COMPLETE, TextureManager addtexture. All fire correctly.
- **Shared loader queue pollution**: Plausible â€” `keyExists` silently drops duplicates. Mitigated by dedicated loader. Did NOT solve.
- **Promise hangs forever (variant=null branch)**: Fixed. Promise always resolves now.
- **shouldUpdateSprite false on plain species change**: Patched. Forces re-load when displayed species drifts.
- **Animation key missing crashes `.play()`**: Phaser's `.play()` with missing key just logs a warning and stays on prior frame â€” no crash, but explains "wrong sprite" if a later play() with valid key never happens.
- **Audio 404 for ER customs**: Ruled out. `getCryKey` does `%2000` so ER customs map to vanilla cry keys. Audio doesn't 404.

## Hypotheses NOT yet exhaustively tested (good places to dig)

1. **`pokemonSprite.setVisible(false)` at the start of EVERY `setSpeciesDetails`** (line 4302) â€” every rapid click hides the sprite. If the LAST click's load fails for any reason, the sprite STAYS HIDDEN. The debounce makes this worse: during a fast scroll, the sprite is hidden constantly until the user pauses 80ms+. If the user reports "blank sprite" they may be seeing this hidden state, not a load failure. **CHECK**: is `setVisible(true)` actually being called for the final click? Add a console.log in the .then() to verify.

2. **The dedicated loader's `BOOT` event miss** â€” `LoaderPlugin` constructor hooks `scene.sys.events.once(BOOT, this.boot)`. By the time we create one in `loadAssets`, the scene has long-since booted. The `boot()` method (which registers `DESTROY â†’ this.destroy` cleanup) NEVER fires for our dedicated loaders. They keep references to scene/systems/textures even after `destroy()` is called manually. **CHECK**: do these leak listeners across many rapid clicks? Use Chrome DevTools Memory tab.

3. **PromiseMicrotask scheduling**: when 50 dedicated loaders each fire COMPLETE in the same frame, all their `.then(() => buildAnim())` callbacks queue as microtasks. Each `buildAnim` calls `globalScene.anims.generateFrameNames` which iterates 400 frame names + `console.warn` suppression overhead. 50 Ă— 400 = 20,000 generated frame names per frame. Could starve the browser event loop briefly, but shouldn't permanently break loading.

4. **`speciesLoaded` map gets stale**: `this.speciesLoaded.set(species.speciesId, true)` runs in .then() even if cancelled? Let me check â€” actually it's GUARDED by `if (assetLoadCancelled.value) return;` so only fires on success. BUT â€” if a load "succeeds" in the dedicated-loader path (COMPLETE fires) but the texture didn't actually land in cache (settle's existence check is false â†’ resolve without finalize), then speciesLoaded is NOT set. Future logic that depends on speciesLoaded may misbehave. **CHECK line 3064**: `if (this.speciesLoaded.get(species.speciesId) || randomSelection) { ... .cry() }`. What other code reads speciesLoaded?

5. **The `pokemonSprite` GameObject sharing state across renders**: `this.pokemonSprite = globalScene.add.sprite(53, 63, "pkmn__sub")` (line 703) is created ONCE. Every `.play(key)` swaps its animation. If two .then() callbacks race to call .play() in quick succession (e.g. a stale load completes after a fresh load already played), the LAST one wins regardless of which species is selected. The `assetLoadCancelled` guard should prevent this, BUT â€” what if the guard check happens before the play() but a microtask race causes the cancelled play to fire anyway? **CHECK**: is there a way to verify which species the sprite is currently animating? Compare `pokemonSprite.anims.currentAnim?.key` to the selected species's spriteKey.

6. **HMR (Vite hot module replacement)** â€” does it preserve `pokemonSprite` state across reloads in dev? Could explain why "after a while" things break â€” class instances may be stale.

7. **The `assetLoadCancelled` token + debounce timer interaction**: my debounce code captures `assetLoadCancelled` in the closure. When user clicks RAPIDLY but each click is ~95ms apart (just over 80ms debounce), the timer FIRES for each click, but the .then() for click N-1 may not have completed when click N happens. Inside .then() for N-1: `if (assetLoadCancelled.value) return` â€” but `assetLoadCancelled` here is N-1's, which is set to true by N's setSpeciesDetails. So N-1 bails. Good. Then N's timer fires, loadAssets called. N completes, plays sprite. Good. **BUT** â€” what if multiple .then()s for the SAME species ran concurrently? E.g., the dedicated loader's COMPLETE fired twice (LOAD_COMPLETE is fake, but COMPLETE could fire from POST_PROCESS phase). Then play() runs twice. Race.

8. **Vite dev server file watcher**: under heavy rapid clicking, does HMR trigger any reloads that disrupt loaders mid-flight?
